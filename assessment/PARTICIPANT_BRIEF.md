# Participant Brief — 24-Hour Build Challenge

Read this fully before you start. **Every team gets this same task and these same rules.** You are judged only on the published rubric (`JUDGING_RUBRIC.md`) — read it too, so you know exactly what earns points.

---

## The task (same for every team)

> ### Build a working AI-powered document assistant web app.
>
> Users **upload one or more documents** (PDF / image / text). Your app uses an **AI model** to **understand, extract, transform, or converse with** the content, and returns something **genuinely useful** through a **web interface**.
>
> **What** problem you solve, **for whom**, and **how** — that's your choice.

That's the whole task. The problem statement is fixed; the solution is wide open. Some example directions (you are **not** limited to these):

- Chat with your PDF / ask questions about an uploaded document.
- Auto-extract structured fields from invoices, forms, or ID documents.
- Summarize or translate a long document into a one-page brief.
- Convert a messy document into clean, structured output (tables, JSON, a filled template).
- An agent that takes a document + a goal and does multi-step work on it.

Pick something real and scope it so a **thin version actually works** in 24 hours. A small thing that works beats a big thing that's broken.

---

## Rules (identical for all teams)

**Team**
- Team size: **3–4 members** (as set by organizers).

**Time**
- You have **24 hours**, from the announced start to the announced end. Same clock for everyone.

**What you may use**
- **Any** language, framework, database, or hosting.
- **Any** AI provider or model (OpenAI, Anthropic/Claude, open-source, local — your call).
- Open internet, open-source libraries, public APIs, and **AI coding assistants are allowed.**

**What is not allowed**
- Submitting a **pre-built** project, or code written before the window. All work must be done **during** the 24 hours.
- Passing off someone else's finished project as your own. Reusing libraries/starters/boilerplate is fine; cloning a complete solution is not.

**Output is not required to be "finished."**
- If parts don't work, that's OK — you are scored on demonstrated ability and how far/how soundly you got, **not** on a binary "did it fully work." Be honest about what works and what doesn't; honesty is scored, not punished.

---

## How to submit (same format for everyone)

Submit these three things by the deadline:

1. **A Git repository link.**
   - Commit history must show the work happened **within the window**. Commit as you go.

2. **A `README` in the repo** that states:
   - **The problem** you chose and **who** it's for.
   - **How to run it** (setup steps + any env keys needed, using placeholders — do not commit secrets).
   - **What works / what doesn't** — an honest status list.

3. **A demo, ≤ 3 minutes** — a recorded video **or** a live walkthrough.
   - Show the core flow: upload → AI does something → useful result.
   - Briefly explain one key decision you made and why.

---

## How you're scored (summary — full detail in `JUDGING_RUBRIC.md`)

100 points total. Weighted toward execution, AI, and innovation:

| Criterion | Points |
|-----------|-------:|
| Technical execution (full-stack works end-to-end, even if thin) | 25 |
| AI integration quality (meaningful, correct — not a gimmick) | 20 |
| Innovation / creativity | 20 |
| Problem & product thinking | 15 |
| Demo & communication | 10 |
| Code quality & collaboration | 10 |

Judges score **only** these criteria. Hype, polish outside the rubric, or reputation do not count.

**Good luck — build something real.**
